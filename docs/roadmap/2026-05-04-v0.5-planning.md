## Decision 1: Multi-user direction

**The question.** Forest Soul Forge today is single-user, local-
first. Multiple users running on different machines is a thing
operators have asked about. Two shapes available:

- **A — Cloud relay.** Each user runs the daemon locally; a
  small cloud-hosted relay routes auth + cross-user invitations.
  Each user's data stays on their machine. Familiar shape;
  similar to how iMessage works.
- **B — Tauri installer.** Each user runs a packaged app
  (Forest Soul Forge.app) that bundles daemon + frontend. No
  cloud component. Multi-user means multi-machine — each user's
  forge is sovereign.

**My recommendation: B (Tauri installer).** Reasons:

- Aligns with the local-first thesis already baked into ADR-0001
  (audit chain as evidence). A cloud relay weakens the "the
  daemon is the user's tool, not Anthropic's tool" story.
- Distribution is the actual hard part. A cross-platform
  installer is a smaller surface than a hosted relay (which
  needs auth, billing, ops, abuse handling, GDPR, etc.).
- v0.4.0 already proves the daemon runs as a binary in front of
  a local frontend. Tauri wraps that without architectural change.
- Reversible: shipping installer-first doesn't preclude a relay
  later. Shipping relay-first commits us to operating cloud
  infrastructure, which is hard to back out of.

**What's blocked until you decide.** Repo structure (one repo
vs. harness-app + harness-bridge pair). Build tooling
(`dist/build.command` currently produces a zip; Tauri would
replace that). Auth posture (none vs. OAuth-via-relay).

## Decision 2: Customer / vertical thesis

**The question.** Who is the v0.5 user? The v0.4 frontend +
audit chain + governance story can be shaped toward different
markets. Three I see:

- **Regulated.** Healthcare / finance / legal teams with strong
  audit + compliance requirements. The hash-chained JSONL is
  exactly what they need for SOC 2 / HIPAA evidence trails.
  High willingness to pay; long sales cycles; lots of asks
  outside the foundry's core (SSO, RBAC, retention policies).
- **SMB / prosumer.** Solo developers + small teams who want
  agent automation without the LLM-API-key sprawl. Lower price
  point; faster sales; the local-first story sells itself
  ("your data stays on your laptop").
- **Internal-tooling consultancies.** Build agents that operate
  on their clients' infrastructure. The audit chain becomes the
  consultant's deliverable. Deep customization needs.

**My recommendation: SMB / prosumer first.** Reasons:

- Lowest distribution friction. Tauri installer + Stripe + a
  pricing page; no enterprise sales motion required.
- v0.4 already has the right shape: local-first, audit-as-spine,
  scheduler-makes-it-set-and-forget.
- Regulated buyers want what we have AND a bunch of things we
  don't (SSO, audit-log shipping to their SIEM, compliance
  certifications). Going there first means 12 months of meta-
  work before shipping the actual product story.
- Internal-tooling consultancies are a great segment but require
  custom integrations per engagement. Wrong shape for a product;
  right shape for services later.

**What's blocked until you decide.** Marketing copy. Pricing
tiers. The "what does v0.5 specifically add for users" question.
The decision affects every feature priority below.

## Decision 3: Mobile platform

**The question.** Mobile is on the v0.4 app-roadmap from earlier
in this session. Three shapes:

- **A — Tauri 2.0 mobile.** Same codebase as desktop. iOS + Android
  via Tauri's mobile targets (still beta as of 2026-05). The
  daemon runs on the user's laptop; the phone app is a remote
  control + viewer.
- **B — React Native.** Native-feel mobile app, separate codebase
  from the desktop frontend. More polish ceiling; doubles the
  frontend maintenance surface.
- **C — PWA-first.** No mobile app. The frontend is responsive
  enough that the operator opens it on their phone's browser.
  Zero net-new code; coupled to the laptop daemon being
  network-reachable from the phone.

**My recommendation: C → A in that order.** Reasons:

- The v0.4 frontend is vanilla JS — making it responsive is a
  CSS pass, not a rebuild. PWA-first ships in days, not months.
- If C reveals real demand for a native app, A (Tauri 2.0
  mobile) reuses the existing codebase. We don't pay a doubled-
  maintenance tax until users tell us we need to.
- B (React Native) is the most common path and the most
  expensive. It commits to a separate codebase that has to track
  the desktop frontend's feature additions forever. Wait until
  C proves users actually want a phone-native app.

**What's blocked until you decide.** Frontend responsive pass
(small, do anyway). Mobile-network considerations (currently
the daemon binds to 127.0.0.1; mobile access needs a local
network bind or a tunnel). Notification posture (push vs poll).

## Decision 4: Free-tier policy

**The question.** What's free forever vs. what's paid?

- **A — Local-only is free forever.** Run the daemon, use
  agents, your data stays on your machine, no payment. Paid
  tier adds cloud sync / multi-device / hosted models / SLA.
- **B — Free trial, then paid.** Limited use of even local
  features after a trial period. Familiar SaaS shape.
- **C — Open-core.** The daemon + frontend are free + open.
  Paid layer is enterprise features (SSO, RBAC, audit-log
  shipping, support).

**My recommendation: A (local-only is free forever).** Reasons:

- Aligns with the local-first thesis. Charging for "running on
  your own machine" is hostile to the value prop.
- Removes the entire trial-expiration / dark-pattern / billing-
  state-leak surface from the local product. Operators using
  the foundry locally never have a "your trial expired" moment.
- Paid features are then opt-in additions (cloud sync etc.)
  rather than re-enabling things the user already had.
- C is fine but assumes there's an enterprise market. See
  Decision 2 — that's a v1.x question, not a v0.5 question.

**What's blocked until you decide.** Pricing copy. Whether the
daemon's auth surface checks for a license key (today: no).
Whether `start.command` ever errors with "your trial expired".

## Decision 5: Repo branding

**The question.** Today everything's `Forest-Soul-Forge`.
Earlier app-roadmap discussion floated `harness-app` (the
desktop wrapper) + `harness-bridge` (the cloud relay) as a
multi-repo split. Three shapes:

- **A — Single repo (status quo).** Everything in
  `Forest-Soul-Forge`. Add `apps/desktop`, `apps/mobile` if/when
  needed.
- **B — Forge + harness split.** `Forest-Soul-Forge` is the
  daemon + agent runtime. `harness-app` is the user-facing
  desktop product. Conceptually clean; logistically more work.
- **C — Rename the whole thing.** The current name is descriptive
  but cumbersome. A product name would help marketing.

**My recommendation: A for v0.5; revisit at v1.0.** Reasons:

- Multi-repo splits are easy to do later (move directories +
  git history) and hard to undo.
- The harness-app + harness-bridge nomenclature presupposed
  Decision 1 went the relay direction. If we go installer-first,
  the harness-bridge repo never exists, and harness-app is just
  Tauri config in `apps/desktop/`.
- C (rename) is a marketing question that depends on Decision 2
  (who's the user). Premature.

**What's blocked until you decide.** Final name in the README,
the install-page copy, the GitHub org structure.

## Technical backlog (orchestrator-independent)

Ranked by leverage. Items 1-2 are non-blocking — I can do them
without your input on the strategic decisions above. Items 3+
benefit from at least Decision 2 (customer thesis) being made.

### 1. Live verify v0.4.0 against the running daemon

**Status.** Sandbox unit tests prove the v0.4.0 code is correct
in isolation (2177 passing). The daemon currently running on
the host is the OLD pre-Burst-86 build. Without a daemon
restart, none of the scheduler / scenario / persistence / control
endpoint code is actually live.

**The work.** Restart the daemon via `stop.command` +
`start.command`. Hit `GET /healthz` — verify
`schema_version=13`. Hit `GET /scheduler/status` — verify
`registered_runners: ['scenario', 'tool_call']`. Configure a
trivial `tool_call` task in `config/scheduled_tasks.yaml`. Watch
it dispatch in the audit chain. Restart again. Verify state
persists.

**Cost.** 15 minutes. **Value.** Closes the integration-test
gap that the Burst 87 audit flagged. Ship-readiness for any
v0.4.x point release.

**Blocked on.** Operator launching the restart commands (I can
write the verify script; the daemon-restart commands run on
your machine).

### 2. Frontend polish queue

Four items still open from the Burst 87 roadmap (Burst 88 fixed
2 of the 6 items it surveyed; these 4 are the remainder):

- **Live-stream chat updates.** Today `sendTurn` blocks on the
  full LLM round-trip. Burst 86.3 added an elapsed-time toast as
  a stopgap. Real fix is using `/audit/stream` to render agent
  replies as they arrive.
- **Chat tab pagination.** Long rooms load every turn. Need
  windowed loading + scroll-to-fetch.
- **Agents-tab archive flow.** The Archive button works (Burst
  88 verified); the post-archive UX (where does the archived
  agent go?) is undocumented in-frontend.
- **Governance-tab constitution diff viewer.** When voice gets
  regenerated or a constitution patches in flight, no UI shows
  the diff. Operator currently reads the soul_generated/ files.

**Cost.** ~1 burst per item (4 bursts total). **Value.** Polish
that pushes the frontend from "operator usable" to "external-
review ready." Useful regardless of which strategic direction.

### 3. Diagnostics dashboard

Per the Burst 87 roadmap. Latency p50/p95/p99 per tool / agent /
model. Useful observability gap; feeds into any "is the foundry
healthy" question. Schema bump v13 → v14 (new tool_call_metrics
table or computed-from-existing tool_calls). New /diagnostics
endpoint + frontend tab (the existing diagnostics button surfaces
a small modal but not a dashboard).

**Cost.** 2-3 bursts. **Value.** Non-trivially helpful for v0.5
buyers — "show me where time goes." Also feeds the customer
thesis (Decision 2) if we go regulated direction.

### 4. Frontend test suite (zero today)

The frontend has 22 vanilla-JS modules, 8 tabs, 0 tests. Burst
88's Chrome MCP audit found 6 latent UX bugs across 4 bursts;
some would have been caught by even basic DOM-level tests.

**Cost.** Substantial — needs a test runner choice (Playwright?
Vitest? Pytest with playwright-python?). 4-5 bursts to get to
30% coverage.

**Value.** Significant. Ships every burst with confidence the
frontend isn't broken in some mode the unit tests don't catch.

### 5. Observability layer (structured logging)

No structured logging today; the daemon writes plaintext to
stderr. For the regulated customer thesis (Decision 2), this is
table stakes — buyers want JSON-shipped logs. For SMB it's nice-
to-have.

**Cost.** 2 bursts (logging library + JSON formatter + per-route
context propagation).

**Value.** Decision-2-dependent.

### 6. README + onboarding

The README is 561 lines and dense. For a new user landing on the
GitHub repo, the path from "I see this" to "I'm running it" is
unclear. A 5-minute Quickstart + a 30-second video would
materially help. Decision-2-dependent (different audiences need
different onboarding).

## Recommended sequence

If you want to keep moving in v0.5 without resolving the
strategic decisions yet, the productive order is:

1. **Burst 97 — Live verify v0.4.0.** Catches any
   integration-level breakage from the v0.4 arc that the unit
   tests didn't see. Cheap insurance before any v0.5 work
   builds on top.
2. **Burst 98 — Frontend live-stream chat updates.** The most
   valuable frontend polish item; works regardless of strategic
   direction.
3. **Burst 99 — Chat tab pagination.** Same.
4. **Burst 100 — Diagnostics dashboard groundwork** (schema +
   query layer; defer the frontend tab to a follow-up burst).

After ~4 bursts of polish, return to this doc and pick up the
strategic decisions. By then I expect a clearer signal on which
direction is most promising.

If you want to resolve the strategic decisions FIRST, the
productive order is:

1. Decide Multi-user direction (Decision 1) — cloud vs Tauri.
2. Decide Customer thesis (Decision 2) — who's v0.5 for.
3. Everything else falls out from those two.

The other three decisions (mobile, free-tier, branding) can wait
on those. They have more obvious-default answers contingent on
the first two.
