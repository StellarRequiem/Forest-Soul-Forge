## §0 — Hippocratic gate: every change must make the Forge strictly better

**Rule:** before any removal, deletion, or "rip out" action, the change passes this 4-step gate:

1. **Prove harm.** What test fails, what feature breaks, what error surfaces, what user is misled? If "no concrete harm" — don't act.
2. **Prove non-load-bearing.** Grep for imports/references; check `git blame` for "placeholder per ADR-X" intent; scan ADRs for forward references. If any reference is "for v0.3+" — keep it with a comment explaining why it stays.
3. **Prove the alternative is strictly better.** Could we fix in place instead? Could we deprecate-and-migrate instead of delete? Removal is the last resort, not the default.
4. **Only if 1+2+3 all pass:** remove, AND record the removal in CHANGELOG + the relevant ADR + an audit-doc entry so the decision is traceable.

If any of 1+2+3 fails: **leave in place + document why.** Add a comment, an ADR note, or a STATE.md callout so future-us knows this thing is intentional, not abandoned.

This rule applies to every "Cleanup" finding (D-* and M-*) and to the C-1 zombie-tools decision. The roadmap below explicitly cites this gate where relevant.

**Dependencies:** none. This is the foundation pass.

**Risk:** F-3 root-cause might surface deeper static-analysis design issues. If so, time-box at 2 hours; either fix or skip-mark with a `# TODO ADR-0030 follow-on` comment.

**Effort:** ~6 hours sustained.

**Sign-off question (revised under §0 gate):** Three options for the 6 zombie tools, in order of how much we preserve:

- **A) IMPLEMENT** them as small wrappers — keeps the original 5 archetype roles fully functional. Modest effort: each is a 100-200 LoC tool. Pros: zero breakage. Cons: adds 6 new files to maintain.
- **B) SUBSTITUTE** with v0.2 equivalents — update the role kits to reference Phase G tools (`osquery_query` for `packet_query`, `log_scan` for `log_grep`, etc.) where the substitute is strictly an upgrade. Original roles keep working with better tools. Pros: no maintenance debt; uses tools we want anyway. Cons: requires Phase G.1.A and G.1.C to land first.
- **C) DEPRECATE** the original 5 archetype roles cleanly — emit a deprecation warning at birth time, point operators at the equivalent Swarm or SW-track role (e.g., `network_watcher` → `log_lurker`), keep the role definition but mark it `deprecated_in: "v0.2"` in the YAML. Plan removal in v0.3 with a migration note. Pros: no breakage today, clear deprecation path, no rushed implementation. Cons: 5 roles carry a deprecation flag for a release.

My revised read: **B for `dns_lookup` and `log_grep`** (clear v0.2 substitutes exist), **A for `packet_query`, `flow_summary`, `baseline_compare`, `correlation_window`** (no clean substitute; implementing keeps the original roles meaningful). Or **C across the board** if we want to retire the original archetype tier in favor of the more specific Swarm/SW-track roles.

What I will NOT do without your sign-off: rip the YAML entries with no migration plan.

### Genuinely new — additions to the roadmap

**N.1 — Persona Forge (ADR-0035 candidate, v0.3 target)**
The trait sliders today produce a 64-char DNA + a constitution + a
templated soul.md voice. The review pushes toward a denser,
operator-curated layer on top: a "Persona Forge" that turns
(trait values + role + genre + operator-supplied bio/history/preferences)
into a reusable rich-prompt profile. Concrete components:
- **Wit dial** (sarcasm/dry-humor scalar trait — extends ADR-0001)
- **Proactivity dial** (latency-vs-anticipation tradeoff trait)
- **Long-term user model** (persistent preferences/history surface, lives
  alongside memory subsystem, scope-gated)
- **Persona render template** that maps traits + user model → dense system
  prompt + few-shot examples
- **Concrete "Jarvis preset"** — a specific trait config + persona text
  ("dry wit, maximum helpfulness, zero unnecessary fluff, anticipates
  needs, confirms high-impact actions, remembers history, adapts").
  Operator-friendly preset shippable as `examples/personas/jarvis.yaml`.

**Why v0.3 not v0.2:** the trait + constitution + soul.md pipeline is
already rich enough to ship persona-shaped agents today. The Persona Forge
adds a layer ON TOP — depends on memory subsystem maturation (long-term
user modeling) and the open-web tools (so agents can ground "anticipates
needs" in real signals). Better as a v0.3 ADR than a Phase G item.

**N.2 — Verifier loop pattern (ADR-0036 candidate, v0.3 target)**
The review names it explicitly: "Before high-impact tool calls, have a
'Verifier' agent or security tier review the plan." The Forge has the
primitives (Guardian genre, approval queue, R3 governance pipeline) but
hasn't formalized the *pattern* — i.e., a skill / triune ceremony that
chains an Engineer's plan through a Reviewer's verification step BEFORE
the privileged dispatch goes through. Concrete deliverable:
- New ADR documenting the Verifier-Loop pattern
- Reference skill manifest `examples/skills/verified_action.v1.yaml` —
  Engineer drafts → llm_think → Reviewer evaluates → if pass, dispatch;
  if fail, refuse + memory_write the rejection
- Hooks into existing approval queue + delegate.v1; no new tools needed
- Fits the SW-track triune naturally (Sentinel = code reviewer)

**N.3 — Observability + monitoring + cost tracking (ADR-0037 candidate, v0.3 target)**
The review flags this as a real gap and it is. Today the Forge has rich
audit chain entries and per-call accounting in the registry, but no
**operator-facing observability surface**:
- No latency histograms or P95 dashboards
- No cost tracking aggregations (the registry has tokens_used + cost_usd
  per tool call, but no rollup view)
- No alerting on budget breach / error rate spike / chain anomaly
Concrete v0.3 deliverable:
- ADR-0037 — observability architecture (likely an `/observability/*`
  router + frontend tab pulling from the registry's tool_calls table)
- KPI dashboard: per-agent token spend, per-tool error rate, audit-chain
  growth rate, approval-queue depth
- Optional Prometheus / OpenTelemetry exporters (gated by env var, opt-in)

**N.4 — Eval harness (extends ADR-0023, v0.2-late or v0.3 target)**
Already on the roadmap (Phase F.3) as the "smallest possible benchmark
suite." The review pushes for richer scope: personality consistency,
task success rate, audit compliance, structured user-satisfaction
feedback. The right move:
- Phase F.3 stays as scoped (provider latency + dispatch overhead +
  audit append throughput) — that's the foundation
- ADR-0023 amendment in v0.3 adds the higher-level evals (persona
  consistency under repeated prompts, task-success scoring against
  curated test cases, audit-compliance checker that walks
  `/audit/tail` looking for missing-event signatures)
