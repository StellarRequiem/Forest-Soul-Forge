# ADR-0042 — v0.5 Product Direction

**Status:** Accepted (2026-05-04). Decided by orchestrator (Alex)
in response to the Burst 96 v0.5 planning doc
(`docs/roadmap/2026-05-04-v0.5-planning.md`).

## Context

After v0.4.0 (Burst 95, tag `v0.4.0`) closed the ADR-0041 set-and-
forget orchestrator arc, five strategic decisions were load-bearing
for v0.5 direction:

1. Multi-user direction (cloud relay vs Tauri installer)
2. Customer / vertical thesis (regulated vs SMB vs internal-tooling)
3. Mobile platform (Tauri 2.0 mobile vs React Native vs PWA-first)
4. Free-tier policy (local-only-free vs trial vs open-core)
5. Repo branding (single repo vs forge+harness split vs rename)

The Burst 96 planning doc framed each as a question + options +
recommendation. This ADR records the orchestrator's decisions
and the consequences that fall out of them.

## Decision

The v0.5 product direction is:

| # | Decision | Choice |
|---|---|---|
| 1 | Multi-user direction | **Tauri installer.** Each user runs a packaged app that bundles daemon + frontend. No cloud component. |
| 2 | Customer / vertical thesis | **SMB / prosumer.** Solo developers + small teams. $10-30/mo price point, self-serve. |
| 3 | Mobile platform | **PWA-first → Tauri 2.0 mobile.** Frontend responsive pass first; native shell only if real demand surfaces. |
| 4 | Free-tier policy | **Local-only is free forever.** No trial expiration on the daemon. Paid features are opt-in additions. |
| 5 | Repo branding | **Single repo for v0.5.** Add `apps/desktop` + `apps/mobile` subdirectories if/when needed. Revisit at v1.0. |

These five decisions are coherent — they reinforce each other
around a single thesis: **Forest Soul Forge is a sovereign,
local-first agent foundry that ships as a desktop app for solo
developers.**

## Reasoning

### Multi-user direction (D1) — Tauri installer over cloud relay

Aligns with the local-first thesis already baked into ADR-0001
(audit chain as evidence FOR the operator, not for Anthropic).
A cloud relay weakens that story — even a thin auth-only relay
becomes the chokepoint that says "your data is on your machine
BUT we know you're online." Tauri keeps the operator's machine
the only thing that knows their forge exists.

Distribution is the actual hard problem. A cross-platform signed
installer with auto-update (Tauri's built-in updater) is a
smaller, more bounded surface than a hosted relay (which needs
auth, billing, ops, abuse handling, GDPR, status pages, on-call).
Picking the bounded surface lets v0.5 ship with a small team.

Reversible. Shipping installer-first does not preclude a relay
later when there's evidence multi-machine users actually need
one. Shipping relay-first would commit us to operating cloud
infrastructure for users who haven't materialized yet.

### Customer thesis (D2) — SMB / prosumer

Lowest distribution friction. A Tauri installer + a Stripe
checkout + a pricing page is the entire v0.5 sales motion. No
enterprise procurement, no SOC 2 audit, no sales-engineering
team.

v0.4 already has the right shape for this audience: local-first,
audit-as-spine, set-and-forget scheduler. SMB users want
"automation that doesn't require me to care about
infrastructure." That's exactly what we have.

The regulated buyer wants what we have AND a long list of things
we don't (SSO, RBAC, SIEM-shipped audit logs, certifications,
contractual SLAs). Going there first means 12 months of meta-
work before anyone uses the actual product. SMB-first means
someone is using the product within weeks of v0.5.

Internal-tooling consultancies are a great segment, but require
custom integrations per engagement. Wrong shape for a product;
right shape for services later.

### Mobile (D3) — PWA-first → Tauri 2.0

The v0.4 frontend is vanilla JS. Making it responsive enough
for a phone browser is a CSS pass, not a rebuild. PWA-first
ships in days, not months — and tells us whether mobile-native
matters before we pay the doubled-codebase cost.

If PWA reveals real demand, Tauri 2.0 mobile reuses the
existing codebase via a thin native shell. We don't pay
maintenance tax on a separate React Native codebase until
users tell us we need to.

### Free-tier policy (D4) — local-only-free-forever

Aligns with D1 + D2. Charging users for "running the daemon
on their own machine" is hostile to the local-first value
prop AND to the SMB price-sensitive segment.

Removes the entire trial-expiration / dark-pattern / billing-
state-leak surface from the local product. Operators using
the foundry locally never have a "your trial expired" moment.
This is a meaningful trust signal — the daemon is genuinely
the operator's tool, not a hostage.

Paid features are then opt-in additions on top of a permanently-
free base. Possible v0.5+ paid surface (none committed in
this ADR): hosted models, cloud sync between machines (Decision
1's deferred relay), priority support, scenario marketplace,
team-shared agent registries.

### Repo branding (D5) — single repo for v0.5

Multi-repo splits are easy later (move directories + git
history) and hard to undo. The `harness-app` + `harness-bridge`
split discussed earlier presupposed Decision 1 went the relay
direction. With D1 = installer, the harness-bridge repo never
exists, and harness-app reduces to Tauri config in
`apps/desktop/` inside the existing repo.

Rename (the third option) is a marketing question that depends
on Decision 2's audience profile firming up. Premature.

## Architecture

### Repo structure (v0.5 onwards)

```
Forest-Soul-Forge/
├── src/forest_soul_forge/      # daemon + agent runtime (unchanged)
├── frontend/                   # browser frontend (unchanged)
├── apps/
│   └── desktop/                # NEW — Tauri shell for v0.5
│       ├── tauri.conf.json
│       ├── src-tauri/          # Rust shell
│       │   └── src/main.rs     # spawns the bundled daemon binary
│       └── ... (Tauri standard layout)
├── config/                     # unchanged
├── docs/                       # unchanged
├── tests/                      # unchanged
└── ... (existing top-level)
```

The Tauri shell does three things:
1. Bundles the daemon (compiled to a single binary via PyOxidizer
   or pyinstaller) inside the app package.
2. Spawns the daemon on app launch; binds to a random localhost
   port; passes the URL to the frontend's `?api=` parameter.
3. Stops the daemon on app quit. Auto-updater wired via Tauri's
   built-in mechanism.

### Daemon distribution

Today: `dist/build.command` produces a zip via `git archive`.
v0.5: Tauri's bundler produces signed `.app` (macOS), `.msi`
(Windows), `.deb` / `.AppImage` (Linux). The zip path stays for
developers wanting to run from source.

Auto-update: Tauri's update flow signs releases with a key the
operator's installer trusts. v0.5 updates flow as
`Forest-Soul-Forge.app/.bundle/Updater` checking a manifest
URL we control.

### Pricing surface

NOT committed in this ADR. Recorded here as the post-v0.5 work
contingent on D2:

- Free tier: local daemon, all features, forever.
- Paid tier (when shipped): TBD. Candidates: hosted-model
  fallback, multi-machine sync, priority support, scenario
  marketplace.

The paid tier is gated on us actually shipping v0.5 first and
seeing what users ask for. Pre-committing pricing now would
violate the "decisions need orchestrator input + signal from
real users" discipline that drove this ADR.

### Mobile execution

Phase 1 (v0.5): CSS responsive pass on the existing frontend.
The `frontend/css/style.css` file gets media queries; the
8 tabs become tab-bar-friendly on narrow viewports. No native
build.

Phase 2 (v0.5+, contingent on demand): Tauri 2.0 mobile
targets reuse the same `apps/desktop` shell with target-specific
config. Daemon would live on the user's laptop; phone app
connects via local network or Tailscale. This is the part D3's
deferral leaves room for.

## Tranche plan

| Tranche | Scope | Burst |
|---|---|---|
| **T1** | This ADR (decision record) | 97 (this) |
| **T2** | Frontend responsive pass (CSS-only) — closes the PWA-first half of D3 | 98 |
| **T3** | Tauri shell scaffolding — `apps/desktop/` with Tauri config that spawns the daemon binary | 99-100 |
| **T4** | Daemon binary build — PyOxidizer or pyinstaller integration into `dist/build.command` | 101 |
| **T5** | End-to-end Tauri build producing a signed .app — auto-updater wired | 102-103 |
| **T6** | Pricing/landing page groundwork (separate from the daemon) — not in this repo's scope | post-v0.5 |

T2-T5 are sandbox-friendly (sandbox can write Tauri config
files; building the actual binary requires a host-side step the
operator runs). T6 is marketing scope and lives outside this
ADR.

## Consequences

**Positive.**

- Coherent product direction. Five decisions reinforce one
  thesis instead of pulling in different directions.
- Lowest-friction v0.5 distribution path. Tauri installer +
  Stripe checkout + pricing page is shippable by a small team.
- Local-first trust story stays intact. Operators don't have
  to take Anthropic's word for "your data is yours" — they can
  see the daemon is the only network endpoint that knows
  anything.
- Frontend responsive pass is value-add regardless of mobile
  direction (operators on laptops use narrow windows too).
- Repo structure stays simple. New code lives in `apps/desktop/`
  rather than spawning a sibling repo.

**Negative.**

- Tauri's mobile support is still beta as of 2026-05. PWA-first
  buys time but doesn't eliminate the bet.
- Daemon-as-binary distribution is a non-trivial new path. Today
  the daemon assumes Python 3.11 is installed on the host; the
  Tauri shell needs to bundle a Python runtime or replace it
  with a compiled binary. PyOxidizer adds build complexity;
  pyinstaller has size implications (50-100MB per bundle).
- Free-forever local tier means the ARPU floor is the paid-tier
  conversion rate. Need to ship enough paid value to cross-
  subsidize free users.
- No cloud relay means no v0.5 multi-machine sync. Users with
  laptop+desktop will ask for it; the answer for v0.5 is "use
  Tailscale or sync via git on a private repo." Acceptable for
  the SMB segment but a real friction point.
- Single-repo structure means the daemon's release cadence is
  coupled to the desktop app's release cadence. A daemon hotfix
  ships through the Tauri updater; that's slower than just
  re-running `pip install -e .`.

## Open questions (deferred to later ADRs)

- **Daemon-as-binary build tool.** PyOxidizer vs pyinstaller vs
  Nuitka. Each has tradeoffs (size, build complexity, stdlib
  coverage). Decided in Burst 101 when the binary actually
  needs to land.
- **Code signing infrastructure.** macOS notarization needs an
  Apple Developer account ($99/yr). Windows code-signing needs
  a cert ($200-500/yr). Decided when we go to ship.
- **Auto-updater manifest hosting.** Tauri needs a JSON manifest
  served from a URL. Could be GitHub Releases (free), a CDN
  (S3 + CloudFront), or a custom server. Decided in Burst 102-103.
- **Pricing model details.** What goes in the paid tier? Decided
  post-v0.5 once usage signal exists.
